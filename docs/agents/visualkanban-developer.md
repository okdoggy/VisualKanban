# VisualKanban Developer Agent Playbook

이 문서는 `/prompts:visualkanban-developer` Agent의 기준 문서입니다.

## 목적
- VisualKanban 구현 과정에서 얻은 실전 노하우를 재사용
- 다음 프로젝트에서 반복 피드백을 사전에 차단
- UI/UX, 권한, 동기화, 운영 안정성 관점의 조언 제공

## 자주 받았던 피드백 요약
1. **UI 중복/불필요 박스가 많다**
   - 해결: 핵심 편집 요소만 남기고 상단/중복 박스 제거
2. **상세 팝업이 복잡하고 정보 밀도가 낮다**
   - 해결: 제목 직접 편집 + 핵심 필드 한 줄 배치 + 불필요 설명 제거
3. **드래그 UX가 직관적이지 않다**
   - 해결: 핸들 아이콘 의존 제거, 텍스트/행/그래프 직접 드래그 지원
4. **검색/사용자 선택 UX가 대규모 인원에 불리하다**
   - 해결: 자동완성 입력 기반으로 통일
5. **테마 가독성 문제**
   - 해결: 모던 라이트/브라운은 그림자 최소화, 대비 중심으로 재설계
6. **로그인/세션/동기화 회귀**
   - 해결: 세션 보호 로직 + 동기화 충돌 방어 + seed revision 기반 리셋

## UI/UX 구현 원칙
- 사용자가 “지금 수정해야 할 것”만 보이게 한다.
- 같은 개념은 페이지별 위치와 형식을 맞춘다.
- 강조는 테두리보다 내부 하이라이트가 우선이다.
- 설명 문구보다 즉시 편집 가능한 구조를 우선한다.

## 권한/데이터 원칙
- 프로젝트 참여 전제 권한: owner/write/read
- 참여자만 삭제/편집 가능한 영역 명확화
- 동시 사용자는 optimistic concurrency + 충돌 재동기화로 보호

## 개발 시 최종 체크리스트
- [ ] typecheck/lint/build 모두 통과
- [ ] 로그인/비밀번호변경/권한가드 회귀 없음
- [ ] 상세 팝업에서 중복 박스/중복 정보 제거 유지
- [ ] 자동완성 UX가 칸반/간트/멤버 관리에 일관 적용
- [ ] 다크/모던 계열에서 텍스트 대비(가독성) 확보

## 다음 프로젝트에 주는 조언
- 기능 추가보다 **정보 구조 단순화**를 먼저 설계하라.
- 대규모 사용자 환경을 가정하면 선택 UI는 초기에 자동완성으로 설계하라.
- 인터랙션(드래그/편집)은 “보여지는 그대로 조작” 원칙을 지켜라.
- 초기부터 회귀가 자주 나는 인증/권한/동기화에 테스트 시나리오를 고정해라.
